## Project status

This project was built incrementally, one major component at a time. The original design specification document (`Real_Time Log Monitoring and Anomaly Detection System.txt`) has since been removed from the repo; this file is now the authoritative, up-to-date reference for responsibilities, data flow, and database columns — kept in sync with the implementation as it evolves. See **Milestones** below for what's implemented vs. still pending.

**Current phase**: the data/model retraining work is complete (two full retrain cycles run, compared, and analyzed — see Isolation Forest retraining notes below). **Further "just bigger" Isolation Forest retraining is explicitly not recommended** — a ~13x larger, deliberately more diverse second retrain produced no measurable improvement over the first, because the bottleneck is the mismatch between training-window sizes and deployment/evaluation-window sizes, not data volume (see Evaluation harness below for the evidence). An engineering-polish pass is also complete: DemoApi healthcheck, TrafficSimulator readiness waiting, a window-level anomaly evaluation short-circuit, safer SIGTERM handling in LogProcessor, unit tests for the four pure detection modules, and a human-facing `README.md`. No detection algorithm, dedup strategy, or architectural boundary changed during either of those passes.

**This phase: answering Research Question 3** ("How does stream processing compare to batch processing in terms of detection latency and responsiveness?"), the one research question from the original proposal that had no implementation work behind it at all until now — see `Tarik_Bilgin_Demirci_Thesis_BriefNote.pdf` for the full RQ list. **RQ3 is now answered.** The approach is a **replay-based batch simulation for evaluation purposes only** (`src/LogProcessor/batch_simulator.py` + `ml/evaluation/run_batch_comparison.py`) — explicitly **not** a second production pipeline: it reuses the same four detection modules (`anomaly_rules.py`, `dynamic_baseline.py`, `isolation_forest_detector.py`, `hybrid_decision.py`) completely unchanged, reads already-persisted `logs_raw`/`logs_agg` data, and only changes *when* detection runs (fixed intervals instead of immediately per message). The live streaming pipeline (RabbitMQ consumer, `database.py`'s per-message transaction) is untouched by this phase — see "Batch simulation (RQ3)" below for the full design and findings. With this, **all 5 of the original research questions now have real, evidence-based answers** — RQ3's answer being that stream wins decisively on latency (sub-30ms vs. an interval-bounded delay of tens of seconds to minutes), while batch's complete-window context measurably changes a subset of detections rather than merely delaying them.

## Architecture

Event-driven pipeline, all services run locally via Docker Compose (no cloud deployment in scope):

```
TrafficSimulator (Python) --HTTP--> DemoApi (.NET) --structured logs--> RabbitMQ --messages--> LogProcessor (Python) --raw logs/aggregations/anomalies--> PostgreSQL --> Grafana
```

End-to-end, today: an HTTP request hits DemoApi → DemoApi logs structured JSON to stdout AND publishes the same `RequestLogEntry` to RabbitMQ's `logs.raw` queue → LogProcessor consumes it, and in **one Postgres transaction**: inserts into `logs_raw`, upserts the 1-minute aggregation window in `logs_agg`, runs rule-based anomaly checks against both the event and the freshly-updated window, inserts any anomalies into `anomalies`, then commits once and acks the RabbitMQ message. Any failure at any stage rolls back the whole transaction and nacks/requeues the message — nothing is ever partially committed.

Key architectural rules (violating these contradicts the design doc):
- **DemoApi** has no database of its own — it only produces structured logs for endpoints `POST /api/login`, `GET /api/products`, `POST /api/cart/add`, `POST /api/payment/checkout`. It supports a `X-Simulate-Latency: true` header to simulate slow/timeout responses.
- **TrafficSimulator** never writes logs directly — it only calls DemoApi over HTTP to generate normal traffic and controlled anomaly scenarios (`normal`/`latency`/`errors`/`mixed` modes). This keeps logs realistic since they're produced by the real API, not synthesized.
- **RabbitMQ** is transport only, never storage.
- **LogProcessor** is a continuously-running Python background worker, not a web API. It owns: consuming from RabbitMQ, persisting raw logs, window-based aggregation, rule-based anomaly detection, Isolation Forest scoring, and (eventually) periodic model retraining (currently a manual script, not scheduled).

## RabbitMQ setup

- `rabbitmq:3.13-management` image — broker on `5672`, management UI on `15672` (guest/guest, `http://localhost:15672` locally).
- Single queue: **`logs.raw`**, durable, default exchange (routing key = queue name). DemoApi publishes, LogProcessor consumes with manual ack and `prefetch_count=1`.
- LogProcessor's consumer (`src/LogProcessor/rabbitmq_consumer.py`) reconnects with a fixed delay (`RECONNECT_DELAY_SECONDS`, default 5s) on connection loss, and on a per-message processing failure nacks with `requeue=True` after a short backoff (`PROCESSING_RETRY_DELAY_SECONDS`, default 2s) to avoid a tight retry loop — discovered this was necessary when a Postgres outage caused ~22,000 retries in 20 seconds before the backoff was added.

- **Endpoints** (`Controllers/`): `LoginController`, `ProductsController` (fixed 5-item catalog), `CartController`, `PaymentController`. All deterministic — no randomness, fixed validation rules (`quantity <= 0` → 400, unknown `productId` → 404, `amount <= 0` → 400, missing login fields → 400).
- **Structured JSON logging**: `Middleware/RequestLoggingMiddleware.cs` wraps every request, honors `X-Simulate-Latency: true` (fixed 2000ms delay), builds a `Logging/RequestLogEntry.cs` (`timestamp, endpoint, method, statusCode, userId, responseTimeMs, requestId, simulatedLatencyMs`), camelCase JSON to stdout.

Settings from `RabbitMqOptions` bound to `"RabbitMq"` config section; `docker-compose.yml` sets `RabbitMq__Host: rabbitmq`. Publish failures are caught/logged, never thrown — RabbitMQ being down doesn't break the HTTP response.
- **`GET /health`** (added this phase, in `Program.cs`, no controller needed): returns `200 {"status":"healthy"}`. Deliberately bypassed by `RequestLoggingMiddleware` (an early-return path checking `context.Request.Path`) so Docker's healthcheck polling every 5s, indefinitely, never inflates `logs_raw`/`logs_agg` with a synthetic `/health` endpoint or trips anomaly detection — verified: `logs_raw` has exactly the 4 real endpoints and none named `/health` even after sustained polling. Backs the `demoapi` Docker healthcheck (`curl -f http://localhost:8080/health`, `curl` added to the runtime image in `Dockerfile` for this purpose only) and is polled directly by TrafficSimulator's own readiness wait (see TrafficSimulator implementation status below).

## LogProcessor implementation status

Python background worker at `src/LogProcessor/`, module layout:
- **`rabbitmq_consumer.py`** — connects, declares `logs.raw`, consumes with manual ack; reconnect-on-loss and backoff-before-requeue as described above. **SIGTERM handling (reworked this phase)**: `request_shutdown()` only sets a module-level flag — safe to call from a signal handler, since it never touches pika/psycopg2 objects directly (not guaranteed re-entrant-safe if called mid-interruption of an unrelated call). The flag is checked exactly once per message, inside `_callback`, immediately *after* that message has been fully acked or nacked — never before, and never mid-processing — at which point `ch.stop_consuming()` is called from that safe, normal callback context (not from the signal handler itself). This replaced an earlier approach where the handler raised `SystemExit` directly, which could in principle interrupt a psycopg2/pika call at an arbitrary point. Verified both paths: stopping mid-traffic (message in-flight) shows the in-flight message fully persisted and acked, then `"Shutdown requested and current message fully handled"` → `"Consumer stopped gracefully (SIGTERM)"`, exit code 0; stopping while idle (no message arriving) has no safe checkpoint to hit, so Docker's default 10s grace period elapses and SIGKILLs the container (exit code 137) — harmless, since nothing was in-flight to lose, but a known, accepted shutdown-latency trade-off under idle conditions.
- **`database.py`** — `persist_log_entry(entry, window_start, window_end, detected_at)` is the single entry point that does raw insert + agg upsert + anomaly inserts in one transaction, returns the list of anomalies detected (with an `inserted` flag distinguishing a fresh insert/upgrade from a deduped no-op). Computes one `processed_at = datetime.now(timezone.utc)` per call (distinct from `detected_at`, the event's own timestamp) and attaches it to every anomaly row. Request-level and rule-based/dynamic-baseline window-level anomalies still insert via `ON CONFLICT ... DO NOTHING`; `hybrid_anomaly` rows go through a separate `INSERT_HYBRID_ANOMALY_SQL` that instead does `ON CONFLICT ... DO UPDATE ... WHERE EXCLUDED.anomaly_score > anomalies.anomaly_score` — same dedup index, but upgrade-in-place semantics (see Anomaly detection strategy below).
  - **Window-level short-circuit cache (added this phase)**: two in-process, unbounded (but tiny — a handful of entries per window/endpoint) module-level caches, `_resolved_window_anomaly_types` (a set of `(window_start, endpoint, anomaly_type)`) and `_hybrid_max_score_by_window` (a dict keyed by `(window_start, endpoint)`). Once `high_error_rate_window` or *both* `dynamic_baseline_*` types are confirmed resolved for a window (via a real insert or an `ON CONFLICT` no-op — either way proves the DB already has the row), `anomaly_rules.evaluate_window_rules`/`dynamic_baseline.evaluate_window_baseline` are skipped entirely for that window on subsequent qualifying requests — for `dynamic_baseline` this also skips the `SELECT_RECENT_WINDOWS_SQL` query, not just the Python evaluation. `hybrid_decision.evaluate` is skipped only once a window's stored tier reaches `CRITICAL` (the max — cannot be upgraded further). This is a pure performance optimization: the cache starts empty on every process restart, and the database's unique indexes / upgrade-only `WHERE` clause remain the sole correctness authority — a cache miss just costs one extra query/evaluation, never a wrong result.
- **`aggregation.py`** — `compute_window(timestamp_str)` parses the (variable-precision, `Z`-suffixed) ISO timestamp and floors it to a 1-minute bucket, returning `(parsed_timestamp, window_start, window_end)`.
- **`anomaly_rules.py`** — pure functions, no I/O: `evaluate_request_rules(entry, detected_at)` and `evaluate_window_rules(endpoint, detected_at, error_rate, request_count)`. See Anomaly detection below for the actual rules.
- **`ml_features.py`** — `build_feature_vector(status_code, response_time_ms, endpoint, window_request_count, window_error_rate)` builds the fixed-order feature vector consumed by both training and inference. Uses `response_time_ms`, `status_code`, one-hot encoded `endpoint`, `window_request_count`, `window_error_rate` — **not** the design doc's `request_count, error_rate, avg_latency, p95_latency, unique_ip_count, encoded endpoint` list, since `p95_latency`/`unique_ip_count` aren't computed yet (see Database schema below). This substitution is a deliberate stopgap, not yet reconciled with the design doc.
- **`isolation_forest_detector.py`** — `evaluate(entry, detected_at, window_request_count, window_error_rate)`, called from inside `database.py`'s transaction. Lazily loads `models/isolation_forest.pkl` (path from `config.ISOLATION_FOREST_MODEL_PATH`) once; if the file is missing or fails to load, logs a message and returns no anomalies (rule-based detection is unaffected). **Feature validation**: immediately after a successful load, compares `bundle["feature_names"]` against `ml_features.FEATURE_NAMES` with strict equality (order-sensitive); on any mismatch (including a bundle with no `feature_names` key at all) it logs the exact model-vs-current feature lists, discards the bundle, and disables ML-based detection for the rest of the process's lifetime — never scores with a mismatched vector. Verified by deliberately writing a model with altered `feature_names` and confirming zero `isolation_forest_anomaly` rows were produced while `rule_based`/`dynamic_baseline`/`hybrid` continued unaffected. On a validated model, runs `predict`/`decision_function` on the feature vector and emits an `isolation_forest_anomaly` (severity `medium`, `detection_method="isolation_forest"`) when flagged, routed through the same per-request dedup path (`request_id, anomaly_type`) as the rule-based anomalies.
- **`train_isolation_forest.py`** — standalone manual script (`python train_isolation_forest.py`, not wired into `main.py` or Compose): loads all `logs_raw` joined with `logs_agg` window stats, excludes rows whose `(endpoint, window)` already has a recorded anomaly, builds feature vectors via `ml_features.py`, splits 80/20 train/holdout (fixed seed), fits `sklearn.ensemble.IsolationForest(n_estimators=100, contamination="auto")` on the train split only, evaluates on both splits, saves `{model, feature_names, trained_rows, rows_loaded, rows_excluded_anomalous, holdout_rows, contamination, holdout_anomaly_rate}` to `models/isolation_forest.pkl` via `joblib`. Prints rows loaded/excluded/trained/held-out, contamination, and holdout anomaly rate. Retraining is still manual — no scheduled/periodic job exists.
- **`dynamic_baseline.py`** — pure function, no I/O: `evaluate_window_baseline(endpoint, detected_at, current_avg_latency, current_error_rate, historical_windows)`. `database.py` queries the last 10 prior `logs_agg` windows for the endpoint (strictly before the current `window_start`) and passes them in. Requires >= 5 prior windows before evaluating (returns `[]` otherwise — no flagging during cold start). See Anomaly detection strategy below for the formula.
- **`hybrid_decision.py`** — pure function, no I/O: `evaluate(endpoint, detected_at, fired_anomalies)`. Called from `database.py` with the union of whatever rule-based/dynamic-baseline/Isolation Forest anomalies fired in the current pass; returns 0 or 1 `hybrid_anomaly` dict. See Anomaly detection strategy below for the tier table.
- **`config.py`** — all connection settings (RabbitMQ + Postgres), retry delays, and `ISOLATION_FOREST_MODEL_PATH` (default `models/isolation_forest.pkl`), all overridable via env vars.
- **`main.py`** — wires it together: prints the consumed entry, calls `persist_log_entry`, prints persistence confirmation and any detected anomalies (`[INSERTED]` or `[DUPLICATE (skipped)]`), across all four detection methods. `_handle_sigterm` (reworked this phase) now just calls `rabbitmq_consumer.request_shutdown()` — see that module's entry above.
- **Unit tests (added this phase)** — `test_anomaly_rules.py`, `test_dynamic_baseline.py`, `test_hybrid_decision.py`, `test_ml_features.py`, flat alongside the modules they test (matching the project's existing flat-file convention, no subpackages). Standard-library `unittest` only — no pytest, no mocking, no external dependencies, since all four modules under test are pure functions with zero I/O. Run with `python3 -m unittest discover -s . -p "test_*.py"` from `src/LogProcessor/`; works on bare host Python, no Docker needed. 35 tests, covering threshold boundaries (e.g. exactly-at-threshold vs. one-under), each hybrid tier transition, one-hot encoding for every known endpoint plus an unknown one, and rolling-baseline sensitivity to historical variance.

## TrafficSimulator implementation status

Python on-demand CLI at `src/TrafficSimulator/main.py`, run via `docker compose run --rm traffic-simulator [--mode normal|latency|errors|mixed] [--count N] [--delay SECONDS]` (never auto-starts). Dockerfile uses `ENTRYPOINT ["python", "main.py"]` + default `CMD` args so `docker compose run` can pass flags directly.
- Deterministic, no real randomness — cycles through a fixed `USERS` list and `VALID_PRODUCT_IDS`/`PRODUCT_PRICES` table (mirrors DemoApi's catalog) by index, so the same `--count` produces the same request sequence every run.
- **`normal`**: full journey (login → products → cart/add → checkout) per iteration, all valid.
- **`latency`**: same journey, rotates `X-Simulate-Latency: true` across one of the 4 endpoints per iteration.
- **`errors`**: rotates through 3 invalid-request generators (missing login password, invalid cart add — unknown product or zero quantity, zero-amount checkout) using an **independent call counter** — not `i % 3` directly, since in `mixed` mode that collided with the mode dispatcher's own `i % 3` and always picked the same error type (found and fixed during implementation).
- **`mixed`**: round-robins `normal`/`latency`/`errors` every 3rd iteration.
- **Readiness wait (added this phase)**: `wait_for_demo_api()` polls `GET {DEMO_API_URL}/health` (bounded: up to 30s, 0.5s between attempts) before generating any traffic, logging `"DemoApi is ready"` once it responds. Belt-and-suspenders alongside the primary fix, `docker-compose.yml`'s `traffic-simulator: depends_on: demoapi: condition: service_healthy` (see Docker infrastructure above) — under normal `docker compose run` usage the container doesn't even start until DemoApi's healthcheck passes, so this poll typically succeeds on its first attempt; it exists mainly to cover running `main.py` directly against a DemoApi not managed by the same Compose stack. A bounded poll, not a fixed sleep — returns as soon as ready rather than always waiting the same amount.

## Evaluation harness

`ml/evaluation/run_evaluation.py` — standalone script (Python stdlib + Docker Compose CLI only, no pip installs needed) that drives 4 controlled TrafficSimulator scenarios (`normal`/`latency`/`errors`/`mixed`) and compares all four detection methods (`rule_based`/`isolation_forest`/`dynamic_baseline`/`hybrid`) on precision/recall/F1 and a time-to-detection proxy. Run with `python3 ml/evaluation/run_evaluation.py` (needs the stack already up).
- **Ground truth is inferred, not measured**: TrafficSimulator never learns DemoApi's internal `request_id` (never returned in HTTP responses), so the harness can't match a specific detected anomaly to the specific request that caused it. Instead it mirrors TrafficSimulator's own deterministic per-iteration dispatch logic to compute an *expected* anomaly count per scenario, then derives TP/FP/FN as `TP=min(detected,expected)`, `FP=max(detected-expected,0)`, `FN=max(expected-detected,0)` — a count-based approximation, documented as such in the generated report.
- Writes timestamped `ml/evaluation/results/evaluation_<ts>.md` and `.csv` (gitignored — regenerate on demand, don't commit).
- Observed pattern across runs (structural, not a one-off fluke): `rule_based` is highest-precision but can't see plain 4xx errors at all (its only status-code rule is `>=500`, which DemoApi never returns); `isolation_forest` is highest-recall but noisiest (fires on plain `normal`-scenario traffic too); `hybrid` lands in between, meaningfully more precise than `isolation_forest` alone; `dynamic_baseline` recall is sensitive to scenario order within a run, since its rolling baseline carries history across scenarios (by design — see Anomaly detection strategy above).
- **Three-way comparison so far** (original baseline model → retrain 1, large-but-narrow dataset → retrain 2, larger-and-more-diverse dataset; see Isolation Forest retraining notes below for how each dataset was built):

  | Method | Metric | Baseline | Retrain 1 | Retrain 2 |
  |---|---|---|---|---|
  | `rule_based` | P / R / F1 | 1.00 / 0.54 / 0.70 | 1.00 / 0.54 / 0.70 | 1.00 / 0.54 / 0.70 |
  | `dynamic_baseline` | P / R / F1 | 1.00 / 0.08 / 0.15 | 1.00 / 0.08 / 0.15 | 1.00 / 0.04 / 0.08 |
  | `isolation_forest` | P / R / F1 | 0.36 / 1.00 / 0.53 | 0.20 / 1.00 / 0.33 | 0.20 / 1.00 / 0.33 |
  | `hybrid` | P / R / F1 | 0.57 / 0.17 / 0.26 | 0.64 / 0.29 / 0.40 | 0.67 / 0.33 / 0.44 |

  `rule_based` never moves (doesn't touch the model — a standing sanity check the change is correctly isolated). `isolation_forest` between retrain 1 and retrain 2 is **identical down to the individual scenario** (same detected/FP/TP in all 4 scenarios, not just the same aggregate) despite training on ~13x more clean rows (192 → 2,480) — see the retraining note below for why more volume alone didn't help. `dynamic_baseline` recall dropped slightly (0.08→0.04) — expected, its rolling baseline is influenced by all accumulated `logs_agg` history including retrain 2's much larger/higher-volume windows, which raised the threshold further. `hybrid` keeps improving on every metric across all three points (0.26→0.40→0.44 F1) purely because its agreement requirement filters a growing fraction of `isolation_forest`'s (unchanged) noise while `rule_based`/`dynamic_baseline` keep contributing real signal — the hybrid layer's relative value keeps growing precisely because the ML layer hasn't gotten any better.

## Batch simulation (RQ3)

Answers the proposal's RQ3 ("How does stream processing compare to batch processing in terms of detection latency and responsiveness?") without building a second production pipeline. Two new files, both purely additive — nothing in the live streaming path (`main.py`, `rabbitmq_consumer.py`, `database.py`, or any of the four detection modules) was modified:

- **`src/LogProcessor/batch_simulator.py`** — a standalone replay engine, run via `docker compose exec logprocessor python batch_simulator.py --start <ISO> --end <ISO> --interval-minutes 5` (same "manual script inside the container" convention as `train_isolation_forest.py`). Reads `logs_raw`/`logs_agg` directly (already written by the unchanged live pipeline), groups raw rows into 1-minute windows, and imports `anomaly_rules.py`/`dynamic_baseline.py`/`isolation_forest_detector.py`/`hybrid_decision.py` **unchanged** to score them — but only once each window has "closed" as of a fixed batch-interval boundary, instead of immediately per message. Never touches RabbitMQ, never writes to `anomalies`; prints a JSON array of what a batch job would have detected (and when) to stdout for the orchestrator to consume.
- **`ml/evaluation/run_batch_comparison.py`** — orchestrator, imports `run_evaluation.py`'s `SCENARIOS`/`run_traffic_simulator` (not duplicated) to drive the same 4 controlled scenarios against the live stream pipeline, then invokes `batch_simulator.py` over the identical `[start, end)` window for a true apples-to-apples comparison. Run with `python3 ml/evaluation/run_batch_comparison.py --interval-minutes 5`.

**Batch execution timing is aligned to a fixed clock grid** (epoch-based N-minute boundaries), not relative to whenever a given evaluation scenario happens to start — this was a real bug caught during validation: an early version computed cutoffs relative to each scenario's own start time, which (since scenarios run in seconds, far shorter than a 5-minute interval) collapsed "batch" down to "detect at the end of the observation window" rather than genuinely simulating a fixed periodic cadence, producing artificially low (~5-25s) batch latencies. Clock-alignment fixed this (batch latencies now correctly land in the tens-of-seconds-to-minutes range, bounded by the interval).

**Latency measurement** (existing columns only, no schema changes): stream = `processed_at - detected_at` (already-stored columns — the same fields documented in Database schema below); batch = simulated `batch_execution_time - detected_at`, where `batch_execution_time` is the next interval boundary after the anomaly's window closed — deterministic, not a real wall-clock wait, since this is a post-hoc replay of already-persisted data.

**Design choices for evaluation-only faithfulness** (all documented in `batch_simulator.py`'s own docstring):
- Window-level rules (`high_error_rate_window`, `dynamic_baseline_*`) and the hybrid decision are evaluated **once per (window, endpoint)** using the window's *final* aggregate — not once per request with a partial aggregate, the way stream mode necessarily does it. This is what a real batch job would naturally do (see all the data at once), and it means batch mode needs no upgrade-in-place/short-circuit machinery at all — it gets the window right in a single pass.
- Request-level rules (`high_latency`, `error_response`, `isolation_forest_anomaly`) still run once per request, exactly as in stream mode — only *when* they run is deferred to batch time.
- `logs_raw` has no `request_id` column, so the simulator uses `f"raw-{logs_raw.id}"` as a synthetic per-request identifier (informational only — the detection functions never branch on its value, just pass it through).

### Results (interval = 5 minutes, 4 scenarios: normal/latency/errors/mixed, count=39 total)

| Mode | Method | Detected | Precision | Recall | F1 | Mean latency |
|---|---|---|---|---|---|---|
| stream | `rule_based` | 13 | 1.00 | 0.54 | 0.70 | 0.00s |
| batch | `rule_based` | 13 | 1.00 | 0.54 | 0.70 | 63.5s |
| stream | `isolation_forest` | 123 | 0.20 | 1.00 | 0.33 | 0.00s |
| batch | `isolation_forest` | 123 | 0.20 | 1.00 | 0.33 | 74.6s |
| stream | `dynamic_baseline` | 1 | 1.00 | 0.04 | 0.08 | 0.00s |
| batch | `dynamic_baseline` | 0 | n/a | 0.00 | n/a | n/a |
| stream | `hybrid` | 12 | 0.67 | 0.33 | 0.44 | 0.03s |
| batch | `hybrid` | 15 | 0.73 | 0.46 | 0.56 | 69.2s |

Every stream-mode number above (detected counts and P/R/F1) is **byte-identical** to the previously-documented Evaluation harness numbers (see table above) — direct confirmation the live pipeline was not touched by this phase, and that measuring latency via `processed_at - detected_at` instead of scenario-relative timing doesn't change which anomalies get counted.

**Why stream is faster**: trivially, by construction — stream mode scores every request in milliseconds inside the same Postgres transaction that persisted it (mean latency 0.00-0.03s, all sub-30ms in practice, since the whole stack runs on localhost with no real network hops); batch mode, by design, must wait for its next fixed interval boundary, so its latency is bounded by (and averages roughly half of) the configured interval — 63-75s observed here against a 5-minute interval, consistent with events landing at varying offsets within that window.

**What changes, what doesn't**: `rule_based` and `isolation_forest` produced **exactly** the same detections in both modes (13/13 and 123/123) — these are purely timing-shifted, not quality-changed, for this dataset. `dynamic_baseline` and `hybrid` **did** differ (1→0 detections, 12→15 detections respectively) — a real, not just cosmetic, detection-quality difference, not only a timing one. The `errors` scenario is the clearest example: stream `hybrid` detected 0, batch detected 3 — because stream mode's hybrid decision only ever sees whatever combination of signals fired on one specific request's own processing pass, while batch mode's once-per-window evaluation sees the *union* of every signal that fired anywhere in the window, letting it catch combinations stream mode's incremental, per-request view missed. The reverse also happened once (`dynamic_baseline` in the `mixed` scenario: stream caught it, batch didn't) — evaluating against the window's fully-settled final aggregate instead of a partial one can push a threshold-sensitive statistic either side of its bar. **Conclusion: batch processing changes both timing and, for signals that depend on how/when partial evidence accumulates (dynamic baseline, hybrid), detection quality too — not just timing.** Purely request-scoped, threshold-based methods (rule-based, Isolation Forest) are timing-only differences here.

**Advantages/disadvantages**:
- Stream: near-instant detection (sub-30ms observed), but each hybrid decision is made on partial, in-progress evidence — occasionally under- or over-counting relative to what full-window context would show.
- Batch: detection latency bounded by (and averaging about half of) the configured interval — for a 5-minute interval, up to several minutes' delay, unacceptable for real-time alerting — but every window-level/hybrid decision is made once, with complete information, which this data shows can genuinely catch combinations a partial, incremental view misses (and occasionally miss ones a partial view happens to catch).

**RQ3 is now answered**: stream processing detects anomalies with negligible latency (sub-30ms here) at the cost of occasionally deciding on partial evidence; simulated batch processing trades that speed for latency bounded by the batch interval, in exchange for complete-window context that measurably changes (not just delays) a subset of detections. Neither is a strict local upgrade over the other on detection quality alone — the real, decisive difference is responsiveness, where streaming wins by multiple orders of magnitude for this workload.

**Reproduce**: `python3 ml/evaluation/run_batch_comparison.py --interval-minutes 5` (stack must already be up). Writes `ml/evaluation/results/batch_comparison_<ts>.md` and `.csv` (gitignored, same convention as `run_evaluation.py`'s own reports).

## Isolation Forest retraining notes

**Retrain 1 (large-but-narrow dataset)**: generating a large, fast-arriving dataset (~1,582 requests across 430 iterations, landing in only ~20 one-minute windows because delays were short) hit a real, working-as-designed edge case: **every one of those 20 windows ended up with at least one recorded anomaly** (partly from the model that was already loaded and scoring in real time as the burst arrived), so the anomalous-window exclusion correctly rejected all 1,582 rows, leaving 0 clean rows to train on. Fixed by temporarily moving the loaded model aside and generating ~206 rows of pure `normal` traffic at a slower pace; retrained on 192/48. That model turned out to generalize poorly (see three-way table above) because those 192 clean rows came from essentially one traffic shape (one to two large, dense windows).

**Retrain 2 (larger, deliberately more diverse dataset, this phase)**: applied the retrain-1 lesson proactively — moved the model aside *before* generating any traffic (not reactively after hitting 0 clean rows), then generated ~727 `normal` iterations across 9 separate batches with varied `--count`/`--delay` specifically to produce `logs_agg` windows spanning a wide `request_count` range (12 to 152 requests/window, vs. one dominant ~200+ window before), plus 200 `mixed` + 50 `latency` + 50 `errors` iterations (70.8/19.5/4.9/4.9% split, matching the 70/20/5/5 target). All new `normal` windows confirmed anomaly-free before retraining (0 contamination, vs. 100% contamination in retrain 1's first attempt). Result: 3,100 clean rows (2,480 train / 620 holdout) — ~13x more training data than retrain 1 — but **the holdout self-flag rate rose to 56.9% (was 33.3% for retrain 1), and live evaluation results against the harness were statistically indistinguishable from retrain 1, scenario-by-scenario.**

**Why more data didn't help (evidence, not speculation)**: querying the actual `window_request_count` distribution of all clean training rows shows it's still bimodal/skewed — ~52% of clean rows come from windows with 101-160 requests, ~38% from 61-100, and only ~3% from windows under 30 requests, even though this phase's batches were deliberately designed to include small windows. The evaluation harness's own scenarios (6-15 iterations) produce windows with request_count typically well under 60. Neither retrain 1 nor retrain 2 has meaningful representation of that range, so both models are effectively scoring evaluation traffic **out of their training distribution** — and produce nearly identical (bad) results because of it. **The lesson: training-set size was never the bottleneck; the mismatch between the training distribution's window-size profile and the deployment/evaluation traffic's window-size profile is.** Fixing this would require deliberately generating a training set whose small-window fraction actually matches or exceeds production-scale traffic, not just adding more large-window volume — a genuinely different data-generation strategy, not a bigger version of the same one.

## Database schema (PostgreSQL)

- **`logs_raw`**: `id, timestamp, endpoint, status_code, response_time_ms, user_id, ip, message` — `ip`/`message` are always `NULL` today since DemoApi doesn't produce them. `RequestLogEntry` (DemoApi) has no `Ip` field at all — IP capture hasn't been started, not just unwired, so `unique_ip_count` can't be computed until that's added upstream.
- **`logs_agg`**: `id, window_start, window_end, endpoint, request_count, error_count, error_rate, avg_latency, max_response_time_ms, p95_latency (nullable, not yet computed), unique_ip_count (always 0, no IP source yet)`. `UNIQUE(window_start, endpoint)` backs the upsert. `error_count`/`avg_latency`/`max_response_time_ms` are maintained via running-aggregate formulas on each upsert (not recomputed from raw rows).
- **`anomalies`**: `id, detected_at, processed_at, window_start, window_end, endpoint, anomaly_type, severity, detection_method, anomaly_score, score_unit, description, request_id (nullable)`.
  - **`detected_at` vs `processed_at`**: `detected_at` is still the *event's own* timestamp (as logged by DemoApi) — unchanged. `processed_at` (added this phase) is `datetime.now(timezone.utc)` captured in `database.py` at the moment LogProcessor actually handles the message, giving a real (if only application-level, not RabbitMQ-queue-time-inclusive) pipeline processing timestamp for the first time. `NOT NULL`, always set — verified 0 NULLs across 1000+ rows after this phase's dataset generation.
  - **`score_unit`** (added this phase, nullable TEXT): what `anomaly_score` actually measures for that row, since it's never been on a comparable scale across methods — `response_time_ms` / `status_code` / `error_rate` (rule-based, depending on rule), `avg_latency_ms` / `error_rate` (dynamic-baseline, depending on which), `isolation_forest_decision_function` (Isolation Forest), `hybrid_tier_rank` (hybrid). Set by whichever detection module produces the anomaly dict; every one of the 7 anomaly-producing call sites across `anomaly_rules.py`/`dynamic_baseline.py`/`isolation_forest_detector.py`/`hybrid_decision.py` sets it — verified 0 NULLs.
  - **Dedup strategy**: two partial unique indexes — `(request_id, anomaly_type) WHERE request_id IS NOT NULL` for per-request anomalies, `(window_start, endpoint, anomaly_type) WHERE request_id IS NULL` for window-level anomalies. Request-level and rule-based/dynamic-baseline window-level inserts use `ON CONFLICT ... DO NOTHING` (unchanged) — a redelivered RabbitMQ message can't double-insert, and a window-level anomaly like `high_error_rate_window` fires **at most once per window** even though the rule re-evaluates on every subsequent qualifying request — verified in testing (4 qualifying requests in one window produced exactly 1 row). **`hybrid_anomaly` is the one exception** (added this phase) — see Anomaly detection strategy below for its upgrade-capable `DO UPDATE`.

## Anomaly detection strategy

**Implemented (rule-based, `anomaly_rules.py`):**
1. `high_latency` — `responseTimeMs >= 1500` → severity `medium`. Per-request.
2. `error_response` — `statusCode >= 500` → severity `high`. Per-request. (DemoApi currently has no path that returns 5xx, so this rule is implemented but not yet exercised by real traffic — only 400/404 are reachable today.)
3. `high_error_rate_window` — the just-updated aggregation window has `error_rate >= 0.5` and `request_count >= 3` → severity `high`. Window-level, evaluated using the `RETURNING` values from the `logs_agg` upsert (no extra query).

**Implemented (Isolation Forest, `isolation_forest_detector.py` + `ml_features.py` + `train_isolation_forest.py`):** scikit-learn `IsolationForest`, trained via a manual script against historical `logs_raw`/`logs_agg` data, persisted with `joblib` at `models/isolation_forest.pkl`, loaded lazily inside `database.py`'s transaction and scored per-request (`isolation_forest_anomaly`, severity `medium`), deduped through the same per-request unique index as the rule-based anomalies. No external ML service — runs in-process inside LogProcessor, per the design doc. **Hardened:** training now excludes rows whose `(endpoint, window)` already has a recorded anomaly (rule-based or ML) before fitting, and does an 80/20 train/holdout split (fixed seed) reporting rows loaded/excluded/trained/held-out, contamination, and holdout anomaly rate — see `train_isolation_forest.py`. **Remaining known limitations:**
- Feature set still substitutes `window_request_count`/`window_error_rate` for the design doc's `p95_latency`/`unique_ip_count` (those columns aren't computed — see Database schema below; documented in a comment in `ml_features.py`).
- Retraining is a manual one-off script run, not scheduled (design doc recommends daily/weekly).

**Implemented (Dynamic Baseline, `dynamic_baseline.py`):** per-endpoint rolling baseline over the most recent `logs_agg` windows (excluding the current one), computed with the stdlib `statistics` module — deterministic, no extra dependency. Flags `dynamic_baseline_latency` (`avg_latency > rolling_mean + 3*rolling_std`, severity `medium`) and `dynamic_baseline_error_rate` (same formula on `error_rate`, severity `high`) once at least 5 prior windows exist for that endpoint. Window-level, stored with `detection_method = 'dynamic_baseline'`, deduped through the existing `(window_start, endpoint, anomaly_type)` partial unique index — the same path `high_error_rate_window` already uses.

**Implemented (Hybrid Decision, `hybrid_decision.py`):** combines whatever rule-based/Isolation Forest/dynamic-baseline signals fired in the *same processing pass* into one `hybrid_anomaly` row (`detection_method = 'hybrid'`). `hybrid_decision.py` itself is unchanged and still pure/stateless — it groups the current pass's fired anomalies by `detection_method`, takes each family's max severity, and maps to a confidence tier: 1 family firing `medium` → `LOW`; 1 firing `high`, or 2 firing `medium` → `MEDIUM`; 2 firing with one `high` (not `rule_based`) → `HIGH`; `rule_based` firing `high` alongside another family, or all 3 families firing → `CRITICAL`. Stored with `severity` = tier (lowercased), `anomaly_score` = tier rank (1–4), description lists contributing families/severities.
- **Upgrade-capable persistence (added this phase)**: unlike other window-level anomalies, `database.py` persists `hybrid_anomaly` via `INSERT_HYBRID_ANOMALY_SQL`, an `ON CONFLICT (window_start, endpoint, anomaly_type) WHERE request_id IS NULL DO UPDATE ... WHERE EXCLUDED.anomaly_score > anomalies.anomaly_score` — same dedup index as before, but now if a *later* message-processing pass in the same window produces a strictly higher tier, the stored row is updated in place (severity, anomaly_score, score_unit, description, detected_at, processed_at all refreshed to the new pass's values); an equal or lower tier leaves the stored row completely untouched (verified: a repeat `CRITICAL` pass correctly left `processed_at` unchanged rather than rewriting it). Verified end-to-end with a real 3-request example: request 1 → `LOW` (`INSERTED`, only `isolation_forest` fired); request 2 to the same endpoint within the same window (via `X-Simulate-Latency`) → `rule_based` + `dynamic_baseline` + `isolation_forest` all fired → stored row upgraded in place to `CRITICAL` (`anomaly_score` 1.0→4.0); request 3 (plain, same window) → only `isolation_forest` fired again → correctly left the row at `CRITICAL`, no downgrade.

**Implemented (Evaluation harness):** see the dedicated section above — compares all four methods quantitatively rather than just qualitatively.
